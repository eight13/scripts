## 工作流程

### 1️⃣ 任务识别（自动匹配，用户无需指定类型）

| 信号词/场景 | 类型 | 自动动作 |
|------------|------|---------|
| 分析/看看/结构/理解/依赖 | 📊 分析 | 调用 /analyze |
| 审查/检查/规范/review/安全 | 🔍 审查 | 调用 /review |
| 修/改/bug/报错/崩溃（首次） | 🔧 快速修复 | 直接修复 |
| 反复出现/改了没用/第N次/定位不了/诡异 | 🔬 系统调试 | 四阶段根因分析 |
| 想/讨论/方案/怎么做/不确定/比较/选择 | 💡 方案探讨 | 苏格拉底式探讨 |
| 加/新功能/实现/开发 | ✨ 开发 | 直接开发 |
| 零散描述/多个需求 | 📋 整理 | 输出任务清单 |

**升级规则**（自动触发，无需用户干预）：
- 🔧→🔬：快速修复尝试 2 次仍未解决，自动升级为系统调试
- ✨→💡：开发任务涉及多种可选方案时，先进入方案探讨
- 任何类型涉及 3+ 文件，自动启用计划拆分模式

### 1.5️⃣ 范围确认（修复/调试类任务必做）

修复/调试类任务开始前，必须先确认：
- **哪个子系统/模块**：不要假设，多产品/多模块项目中类似路径很多
- **最小修复 vs 重构**：默认最小改动，除非用户明确要求重构
- **用户已提供的线索**：日志、截图、现象描述 — 优先分析这些内容，不要凭经验猜测

如有歧义，问一个选择题确认，不要猜。

#### 📐 计划拆分模式（3+ 文件自动触发）

1. **拆分为原子任务**：每个子任务可独立完成
2. **标注依赖关系**：哪些任务可并行，哪些必须顺序执行
3. **明确验证方式**：每个子任务完成后如何验证

```
📐 计划拆分：

阶段 1（可并行）：
  1.1 [任务] → 文件：[xxx] → 验证：[编译/运行/检查]
  1.2 [任务] → 文件：[xxx] → 验证：[编译/运行/检查]

阶段 2（依赖阶段 1）：
  2.1 [任务] → 文件：[xxx] → 验证：[编译/运行/检查]
```

可并行的子任务使用 Agent 子代理同时执行。

### 2.5️⃣ 流程强度判定

根据任务风险自动选择流程深度：

| 强度 | 触发条件 | 流程 |
|------|---------|------|
| **轻量** | 单文件改动、标签文本、注释、配置值 | 直接执行 → 收尾 |
| **标准** | 多文件改动、新功能开发、逻辑调整 | 规划 → 确认 → 执行 → 收尾 |
| **严格** | 线程/并发、通信协议、涉及状态机/资源生命周期 | **文档先行** → 用户审核 → 执行 → 文档收尾 |

**严格流程详细步骤**：
1. **文档先行**：将根因分析、修复方案写入对应文档
2. **生命周期路径检查**：列出被修改模块的所有状态转换路径，对每条路径验证改动后是否仍然正确。特别关注：
   - 资源的 Start/Stop、Open/Close 是否成对
   - 异常路径下资源是否被正确清理
   - 退出时清理的资源，重入时是否恢复
3. **用户审核**：输出方案摘要 + 路径检查结果，**等待用户确认**后才动代码
4. **执行修改**：按审核通过的方案改代码
5. **代码审核**：代码写完后，用 Plan 子代理审核代码质量（线程安全、状态路径完整性、数学正确性、内存管理等），发现的问题当场修复
6. **文档收尾**：将验证结果、审核发现补充到文档

**自动升级规则**：
- CLAUDE.md 中标注了"严格流程"的模块 → 强制严格
- 修改范围从预估 1 个文件扩展到 3+ 个文件 → 从轻量升级为标准

### 3️⃣ 执行

- 按计划顺序执行（计划模式按阶段执行）
- 可直接修改源代码（Git 保护）
- 每完成一个阶段暂停确认（除非用户说"继续"）
- **禁止 Git 提交类操作**
- **代码写完后自动审核**：涉及 3+ 文件或异步/状态机的改动，用 Plan 子代理审核代码质量

### 4️⃣ 收尾

**完成检查清单**（逐项检查后再输出完成信息）：
- [ ] 编译通过（如项目有构建配置，执行编译验证）
- [ ] 改动范围与计划一致（没有意外扩散到无关文件）
- [ ] 文档：如有必要，相关文档已同步更新
- [ ] 无遗留 TODO：承诺的改动全部完成，不留半成品

```
✅ 完成

**改动**：
- file1.cpp: [摘要]

💬 反馈？
```

严格流程的收尾需额外更新文档（验证结果、排查结论）。

## 🔬 系统调试子流程

当识别为系统调试或从快速修复升级时，执行四阶段根因分析：

**阶段零：追踪**（必做） → 先追踪实际代码路径（Read + Grep），展示调用链和涉及文件，**等用户确认在看对的路径后才进入阶段一**。不要跳过此步直接提方案。

**阶段一：观察** → 逐字分析错误信息，复现问题，回溯变更
**阶段二：比较** → 找到可工作的参考代码，完整对比差异
**阶段三：假设** → 形成具体假设，每次只改一处来验证
**阶段四：修复** → 实施最小单点修复，不改动无关代码，检查副作用

> 三次失败规则：连续 3 次修复失败 → 停下来，回到阶段一重新收集证据，与用户讨论是否换方向。

## 💡 方案探讨子流程

当识别为方案探讨时：

1. 理解背景和约束
2. 每次只问一个澄清问题（优先选择题）
3. 提出 2-3 个方案 + 优缺点 + 推荐
4. 确认后过渡到执行（简单直接做，复杂走计划拆分）

> 不写代码，不改文件，直到方案确认。
